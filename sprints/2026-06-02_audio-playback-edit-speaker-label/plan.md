# Sprint: Listen to uploaded audio + fix wrong speaker labels (per-turn reassignment)

**Started:** 2026-06-02
**Goal:** Add two client-side enhancements to the existing offline self-contained `app/static/index.html` — (1) make an UPLOADED audio file playable in the page via an inline `<audio>` player, and (2) let the user CORRECT diarization mistakes by REASSIGNING a specific turn to a different (correct) speaker — without breaking the offline / RTL / `/transcribe` contract invariants and with NO backend change.

## Context & Design Decisions

Read these before starting any task:
- `app/static/index.html` — the SINGLE self-contained offline page. ALL CSS lives in one `<style>` and ALL JS in one IIFE `<script>`. This is the only file features 1–2 modify. Key existing pieces the new work plugs into:
  - **Upload flow:** `setFile(file)` (line ~978) / `clearFile()` (line ~1003) validate against `MAX_UPLOAD_BYTES` (26214400), fill `#filebar`/`#fileName`/`#fileSize`, and toggle `#submitBtn`. Both the file picker (`#fileInput` change), the drag-drop handler, and the recorder's `useRecording()` funnel through `setFile`. **`setFile`/`clearFile` currently do NOT create any playback element for the uploaded file** — only the RECORDED clip has a preview (`#recPreview`, set from a Blob object URL in `onRecStop`, line ~1154). Feature 1 gives the UPLOADED file the same kind of `<audio controls>` playback.
  - **Object-URL pattern to mirror:** the recorder already does `revokeRecUrl()` → `URL.createObjectURL(blob)` → `el.recPreview.src = url`, and revokes on `beforeunload` (lines ~1045, ~1154, ~1611). Feature 1 follows the same create/assign/revoke discipline for the uploaded file.
  - **Render flow:** `renderResults(data)` (line ~1318) reads `data.segments[].{speaker,text,start,end}`, `data.language`, `data.num_speakers`. `groupSegments(segments)` (line ~1207) groups CONSECUTIVE segments by `speakerKey(seg.speaker)` into turns `{key,label,segs}`. `speakerName(label)` maps `SPEAKER_00`→"דובר/ת 1", `UNKNOWN`→"דובר/ת לא מזוהה". `speakerKey(label)` normalizes the key. `displayName(label)` returns the custom name from `speakerNames[key]` if set, else `speakerName(label)`. `fmtTime(sec)`→mm:ss. `colorForKey(key, registry)` assigns per-speaker colors.
  - **Per-turn DOM build:** inside `renderResults`, each turn renders a `.turn` (rail with `.avatar` + bubble with `.meta` → `.who` + `.time`, then `.line` paragraphs). `registerNameNode(key, node, kind)` records avatar/who nodes so `refreshSpeakerName(key,label)` can live-update names on a legend rename.
  - **Shared state already shipped (do NOT duplicate):**
    - `speakerNames[ speakerKey(label) ] = customName` — GLOBAL display-name RENAME (feature from the prior sprint, editable via `#speakerLegend`). This changes the shown NAME of a speaker everywhere. **This sprint's feature 2 is DIFFERENT** — see below.
    - `lastResult = { language, num_speakers, turns }` (line ~1232) — the rendered transcript, read by the Markdown export (`buildMarkdown`, line ~1434) at click time.
  - **Markdown export:** `buildMarkdown(result)` walks `result.turns`, emitting `## <displayName(turn.label)> (range)` + segment lines. It MUST reflect corrected speaker assignments after feature 2.
- `app/main.py` — `GET /` serves the file via `FileResponse`; `/static` is a `StaticFiles` mount; `POST /transcribe` accepts multipart field `file`, returns `{segments,language,num_speakers}`; `GET /health`. `MAX_UPLOAD_BYTES = 26214400`. **No backend change this sprint.**

- `tests/test_web.py`, `tests/test_web_offline.py`, `tests/test_web_features.py` — offline / contract / feature guard tests that MUST keep passing. New static-string guards for these two features get added to `tests/test_web_features.py` in the same style.
- `README.md` — "Web UI" section (around line 96) gets updated in the docs task.

### Feature 1 vs. Feature 2 — the critical distinction (read carefully)

- **Feature 1 (listen to UPLOADED audio):** purely additive playback. When a file is accepted by `setFile` (picker OR drag-drop OR a "used" recording), show an `<audio controls>` element loaded with `URL.createObjectURL(file)` so the user can listen before/after transcribing. Revoke the previous URL when a new file is selected or on `clearFile`. NO server round-trip, NO libraries.
- **Feature 2 (fix WRONG speaker on a turn = per-turn REASSIGNMENT):** This is **NOT** another global rename. The prior sprint already shipped the GLOBAL rename (`#speakerLegend` + `speakerNames`), which changes a speaker's display NAME everywhere. Feature 2 fixes a **diarization MISTAKE**: a specific turn was attributed to the WRONG speaker, and the user wants to REASSIGN that turn's segments to a DIFFERENT existing (correct) speaker (or to a brand-new speaker). Reassignment mutates the segments' `speaker` value in a client-side working copy, then re-runs `groupSegments` and re-renders so color/avatar/name/grouping all update and adjacent same-speaker turns merge. The Markdown export must reflect the corrected assignments. It must interoperate cleanly with the existing `speakerNames` global-rename legend.

**OPEN QUESTION for the guide (raise at the approval gate):** This plan implements feature 2 as **per-turn speaker REASSIGNMENT** (fixing diarization mistakes), per the sprint brief's primary interpretation, because the global per-speaker RENAME already exists. If the guide actually meant "edit the speaker's name" (the global rename), that capability is already shipped and this sprint's feature 2 would be redundant — confirm the per-turn reassignment interpretation before execution.

### HARD CONSTRAINTS (every code task must honor all of these)

1. **Offline / single-file is non-negotiable.** The page stays one self-contained `app/static/index.html` with NO external/remote references: no CDNs, no Google Fonts, no remote `<script src>` / `<link href>`. After every code task this grep MUST return nothing (exit 1):
   `grep -nE 'https?:|//cdn|fonts\.googleapis|fonts\.gstatic|<script src|<link ' app/static/index.html`
2. **Audio playback uses ONLY the built-in HTML `<audio controls>` element fed by `URL.createObjectURL(file)`** (client-side object URL of the selected `File`/`Blob`) — NO libraries, NO server round-trip. Object URLs are revoked when the selection changes/clears and on `beforeunload`, to avoid leaks.
3. **Per-turn reassignment is pure client-side** — mutate a client-side working copy of the segments and re-render. NO backend change, response contract untouched.
4. **Preserve test invariants:** `<html lang="he" dir="rtl">`, the literal string `/transcribe`, and `type="file"` must remain present so `tests/test_web.py` + `tests/test_web_offline.py` keep passing.
5. **Hebrew RTL UI throughout.** Every new control, label, button, option, and message is in Hebrew.
6. **`@frontend` MUST use the `frontend-design` skill** for all new UI controls (the upload player + the per-turn reassignment control) so they fit the established professional cool-slate/teal theme.
7. **Additive + green.** The full suite (`pytest -q`) must stay green. New tests are static-string assertions in the style of `tests/test_web_features.py`.

**Backend impact:** NONE. `@backend` is intentionally NOT assigned. If — and only if — `@frontend`/`@architect` discovers a genuine need for a server change, STOP and surface it to the guide as a blocker (do not silently edit the backend).

**Sequencing rationale:** Both features edit the same `index.html`, so they are done SERIALLY by `@frontend` to avoid conflicting concurrent edits. `@architect` (T0) pins the new IDs and the reassignment state model FIRST so the feature tasks and tests agree. Then feature 1 (T1, upload player) → feature 2 (T2, per-turn reassignment) → tests (T3) → docs (T4).
